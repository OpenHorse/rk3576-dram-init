Chapter 7 Dynamic Memory Interface (DMC)

7.1 Overview

The DMC includes LPDDR4/4X/5 controller (DDRCTL), DDRPHY, DDR monitor and DFI scramble, which are a complete memory interface solution for DDR memory subsystems. The DDRCTL SoC application bus interface supports AXI interface, with flexible address mapper logic allow application-specific mapping of row, column, bank, bank group and rank bits to achieve industry leading high-efficiency, low-latency and high-performance from memory interface. The DDRPHY provides control features to ease the customer implementation of digitally controlled features of the PHY such as DQS clean, read training, write training, dvfs training, periodic training and programmable configuration controls. The DMC supports the following features:  Support LPDDR4/4X/5  Support LPDDR4 protocol up to LPDDR4-4266 speed grade and LPDDR5 protocol up to
LPDDR5-4800  Support up to 2-channel interleaving with a maximum of 16GB capacity in total, 2
ranks for each channel  Support scalable and software-controlled 1:2:2(LPDDR4/4X) or 1:1:2/1:1:4 (LPDDR5)
frequency ratio  Support 5 AXI ports which communicate between the SoC interconnect and DDRCTL  Support 32-bit DDR data bus width for 2-channel interleaving  Support up to 10 kinds of address mapping  Support burst16 for LPDDR4/4X/5  Support different CL/WL latency  Support 16 bank mode and BG mode for LPDDR5  Support open, close, intelligent pre-charge paging policy  Support advance refresh control  Support LPDDR4/4X/5 software controlled frequency change  Support DFI PHY master interface  Support MC-initiated requests and PHY-initiated requests interface  Support DFI data low power and ctrl low power interface  Support auto power-down or auto self-refresh power down when DDRCTL is idle for a
period of time. The DDRCTL's core clock, DDRCTL's AXI clock and interconnect's AXI clock can be gated  Support auto or SW issue entry or exit clock stop/power-down/self-refresh/deep sleep mode  Software programmable Quality-of-Service (QoS) support  Three traffic classes on read commands--high priority reads, variable priority reads,
and low priority reads  Two traffic classes on write commands--normal priority writes and variable priority
writes  Port urgent and port throttling control  Dynamic scheduling inside DDRCTL to optimize bandwidth and latency  Support PMU controlled DDRPHY entry or exit IO retention mode  Support scrambling and descrambling of data written to or read from DDR SDRAM  Support DDR monitor for following features  AMBA 32-bit APB slave interface  Each DDR channel embedded with one DDR monitor module  Support to monitor DDR read or write address within 8GB address range for each
channel  Support to observe whether DDR access address within a specified range  Support to do the following statistics during a programmed number of time through
hardware mode or software mode  Statistics about DDR idle cycles  Statistics about SR/PD/CKSTOP/SRPD/DFI data low power/DFI ctrl low power

7.3 Function Description

7.3.1 DDRCTL
DDRCTL supports 5 AXI Ports, it receives AXI transactions from interconnect directly. These transactions are queued internally and scheduled to the SDRAM while satisfying SDRAM protocol timing requirements. It in turn receives read data on the DFI interface from the DDRPHY block.

Fig. 7-2 DDRCTL Block Diagram DDRCTL contains the following main components.  AXI2HIF block: This block provides the AXI interfaces to system level and HIF interface
to DDRC block. It provides bus protocol handling, data buffering, data bus size conversion and memory burst alignment. Read is stored in a read re-order buffer and return to the AXI Port.  DDRC block: This block issues the read/write commands, carries out the DRAM page management, issues DRAM maintenance commands, and implement the DFI interface. Write data is stored in an SRAM until its associated command is issued to the PHY. Read data is handled by the response engine in the DDRC and is returned on the HIF. To improve DDR performance in different scenarios, there are two modules named urgent_ctrl module and mask_ctrl module outside the DDRCTL.
7.3.1.1 urgent_ctrl
The function of urgent_ctrl module is to control the DDRCTL's awurgent_n/arurgent_n(n=0~4) per port according to the press_n or awqos_n/arqos_n(n=0~4) signals from interconnect module, as well as status signals such as raq_pop_n/waq_pop_n(n=0~4) from DDRCTL. For a given AXI port, there is an embedded queue for each port inside urgent_ctrl module to store arqos or awqos respectively. The queue depth is 12. The arqos or awqos queue becomes urgent if there are commands in the queue and not popped for DDR_GRF_CHA/B_CON16.press_queue_wait_limit*4 cycles. The arqos_out and awqos_out is popped at the same time as DDRCTL internal AR FIFO and AW FIFO. The value of arqos_out, awqos_out and press signals are compared to select the maximum one as the QoS of a given AXI port. Then a port priority comparison module compares the QoS of 5 AXI ports and selects the port which has the largest QoS. The arurgent is generated based on the following conditions.  The port is selected by port priority comparison module.  The arqos queue becomes urgent, arqos_out is largest of awqos_out, press and
DDR_GRF_CHA/B_CON14.press_ar_th. If not, press should be larger than DDR_GRF_CHA/B_CON14.press_ar_th when arvalid is high and arready is low.  When write is blocked by read (the awqos queue becomes urgent inside urgent_ctrl module but read port is granted by PA(Port Arbiter) inside DDRCTL and DDR_GRF_CHA/B_CON16.press_rdwr_order is set 1'b1). The awurgent is generated based on the following conditions.  The port is selected by port priority comparison module.  The awqos queue becomes urgent, awqos_out is largest of arqos_out, press and DDR_GRF_CHA/B_CON14.press_aw_th. If not, press should be larger than DDR_GRF_CHA/B_CON14.press_aw_th when awvalid is high and awready is low.  When read is blocked by write (the arqos queue becomes urgent inside urgent_ctrl module but write port is granted by PA inside DDRCTL and DDR_GRF_CHA/B_CON16.press_rdwr_order is set 1'b1).

7.3.2 DDRPHY
DDRPHY supports LPDDR4/4X/5 SDRAM and provides physical interface solutions for DDRCTL requiring access to JEDEC compatible SDRAM devices. It is optimized for low power and high speed (up to 4266Mbps for LPDDR4/4X and up to 4800Mbps for LPDDR5) applications with robust timing and small silicon area. The DDRPHY has fully digital DLL for 90phase shift of strobe signal. DQS clean, read training, write training, dvfs training and
periodic training are supported. In order to support DDR fast frequency change, there are 2 PLLs in each DDR channel. D0APLL and D0BPLL are used to provide operation clock for DDRPHY channel 0, while D1APLL and D1BPLL are used for DDRPHY channel 1. The output clock of 2 PLLs in each channel are connected to the input of a multiplexer, the output of the multiplexer drives the DDRPHY source clock clk_phy2x. During DDR frequency change period, it just needs to switch to the other PLL that has locked before without waiting for PLL lock.
7.3.3 DFI Scramble
The function of DFI scramble module is to scramble data written to DDR SDRAM and descramble data read from DDR SDRAM. This module is by default bypassed. If it is enabled, it receives data from DDRCTL through DFI interface, scramble written data and then send it to the DDRPHY. It also descrambles date received from DDRPHY and send it back to DDRCTL through DFI interface.
7.3.4 DDR Monitor
The DDR Monitor Module has four functions. The first function is used when debug, it will monitor the DDR read or write address within a specified range. The second function is used to do the statistics about DDR bandwidth and utilization. The third function is used to statistic special pattern for a designated bank or for all banks. The fourth function is used to statistic low power such as self-refresh, power down, self-refresh power down, clock stop and so on.

7.4.3 Detail Registers Description For DDRCTL

Some register fields might use the following optional attributes.

Volatile: If true, indicates in the case of a write followed by read, or in the case of two

consecutive reads, there is no guarantee as to what is returned by the read on the second

transaction or that this return value is consistent with the write or read of the first

transaction. The element implies there is some additional mechanism by which this field

can acquire new values other than by reads/writes/resets. For example, when the controller

updates the register field contents.

Testable: Possible values are unconstrained, untestable, readOnly, writeAsRead, restore.

Untestable means that this field is untestable by a simple automated register test. For

example, the read-write access of the register is controlled by a pin or another register.

Read only means that you should not write to this register; only read from it. This might

apply for a register that modifies the contents of another register.

The Programming Mode Field: The "Programming Mode" field in the description column of

the register tables specifies the type of register. The DDRCTL registers belong to one of the

following types.

 Static: Can be written only when the controller is in reset.

 Dynamic: Can be written at any time during operation.

 Quasi Dynamic: Can be written when the controller is in reset and some specific

conditions outside reset. Please refer to the application notes for detailed information.
